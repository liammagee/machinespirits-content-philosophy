Workflow guide

Fast, Nice-Looking Citations

A practical guide to writing scholarly Markdown with BibTeX, using pandoc & citeproc — built around a real working example, not a feature tour.

For students writing essays, theses, and articles — humanities or otherwise.

1.Why bother with this setup?

If you write essays in Word and manually format your bibliography, you've felt the cost: when a reviewer asks you to switch from APA to Chicago, you reformat by hand. When you reuse a source, you re-type it. When you move a paragraph, in-text citations drift out of order. None of that work has anything to do with thinking.

The Markdown + BibTeX workflow inverts the problem. You write plain prose with citation keys ([@foucault1977preface]) instead of formatted citations, and a small program — pandoc — renders the document with proper citations, a sorted bibliography, and the formatting style of your choice. Switch styles? One flag. Add a new source? One entry. Move a paragraph? Nothing changes.

The setup takes about ten minutes the first time. After that, every essay is faster than the one before it.

Note This guide uses pandoc's built-in --citeproc filter. You don't need LaTeX. You don't need a complex IDE. A plain text editor and a terminal are enough.

2.Install pandoc

Pandoc is a free document converter. Install it once.

# macOS (Homebrew)
brew install pandoc

# Ubuntu / Debian
sudo apt install pandoc

# Windows (winget)
winget install pandoc

Verify it works:

pandoc --version

If the version is 2.11 or later (released 2020), citeproc is built in and you're ready. Earlier versions need a separate pandoc-citeproc package.

3.The .bib file: anatomy of an entry

A .bib file is a plain text file holding your sources. Each source is one entry. Here's a real one from a recent essay:

@book{pasquinelli2023eye,
  title        = {The Eye of the Master: A Social History of Artificial Intelligence},
  author       = {Pasquinelli, Matteo},
  publisher    = {Verso Books},
  address      = {London},
  year         = {2023},
  url          = {https://www.versobooks.com/products/735-the-eye-of-the-master}
}

Three things matter:

  1. The entry type (@book) — tells citeproc how to format this. Common types: @book, @article, @incollection (a chapter in an edited volume), @misc (anything else), @inproceedings (a conference paper).
  2. The citation key (pasquinelli2023eye) — your unique handle. Convention: authoryearword. Use only this in your prose.
  3. The fields — what's required depends on entry type, but author, title, year, and one of (publisher / journal / url) covers most cases.

The most common entry types

TypeUse forRequired fields
@bookA whole bookauthor, title, publisher, year
@articleA journal articleauthor, title, journal, year
@incollectionA chapter in an edited volumeauthor, title, booktitle, editor, publisher, year
@inproceedingsA conference paperauthor, title, booktitle, year
@miscWeb pages, blog posts, videos, anything that doesn't fit cleanlyauthor, title, year, url

Don't memorise these. Look them up when you need them. Most editors (VS Code, Sublime, Vim with plugins) will autocomplete fields once you pick a type.

Tip Almost every academic source page has a "Cite" or "BibTeX" button. arXiv, JSTOR, Google Scholar, ACM, journal websites — they generate the entry for you. Copy, paste, edit the citation key to your liking, done.

4.Citation syntax in the markdown

At the top of your markdown file, add a YAML header pointing to your .bib file:

---
bibliography: dictatorship.bib
---

# My essay title

Here is the body...

In the body, you have two citation forms. Pick whichever reads better.

Parenthetical citation: [@key]

Use square brackets when the citation is parenthetical — i.e., it could be wrapped in (parens) without changing the sentence:

It is Benjamin's notion of mechanical reproduction [@benjamin1969mechanical]
run through silicon chips and linear algebra...
Renders as (Chicago author-date):
It is Benjamin's notion of mechanical reproduction (Benjamin 1969) run through silicon chips and linear algebra…

Narrative citation: @key

Use the bare form when the author is the grammatical subject of your sentence:

The first case is described well in @ouyang2022training, where the model
is conditioned to agree to a minimal set of dos and don'ts.
Renders as:
The first case is described well in Ouyang et al. (2022), where the model is conditioned to agree to a minimal set of dos and don'ts.

Multiple citations

Separate keys with semicolons inside one bracket:

...the history of writing as exomnemonics [@derrida1981dissemination; @stiegler1998technics]

Suppressing the author

If you've already named the author in prose, prefix with - to render only the year:

Foucault [-@foucault1977preface] writes...
Renders as:
Foucault (1977) writes…

Adding page numbers, prefixes, suffixes

Inside the brackets, after the key:

[@kant1993groundwork, p. 31]
[see @kundu2023specific, ch. 2]
[@huang2025valueswild, pp. 14–17]

5.Rendering to HTML, PDF, or Word

One command. Pick your output format by changing the file extension.

# HTML
pandoc essay.md --citeproc -o essay.html

# PDF (requires LaTeX, e.g. MacTeX or TinyTeX)
pandoc essay.md --citeproc -o essay.pdf

# Microsoft Word
pandoc essay.md --citeproc -o essay.docx

The --citeproc flag tells pandoc to process citations using the .bib file you named in the YAML header. It will:

  1. Replace every [@key] and @key with a properly formatted citation.
  2. Append a sorted bibliography at the end of the document, under a heading called "References" (or whatever the locale specifies).
  3. Skip any .bib entries you didn't actually cite — your bibliography only contains what you used.
Tip Want to control where the bibliography appears? Add ## References as the last heading of your markdown. Pandoc places the rendered bibliography under it. If you don't include the heading, pandoc adds one automatically.

6.Choosing a citation style (CSL)

By default, pandoc uses Chicago author-date. To change style, pass a CSL file with --csl:

pandoc essay.md --citeproc --csl=apa.csl -o essay.html

Where do CSL files come from? The Zotero Style Repository has over 10,000 of them, all free. Common ones:

  • chicago-author-date.csl — Chicago 17th, author-date variant
  • chicago-note-bibliography.csl — Chicago 17th, footnote variant
  • apa.csl — APA 7th edition
  • mla.csl — MLA 9th edition
  • harvard-cite-them-right.csl — Harvard (UK universities)
  • ieee.csl — IEEE numeric (engineering)

Download once, save next to your markdown, point pandoc at it. Switching style is now one command-line flag away.

7.A fast workflow that scales

The thing that takes time isn't the citation tooling — it's curating your .bib file. Here's how to keep that fast.

Start a per-project .bib next to your essay

Don't try to maintain one giant .bib across your whole career. Per-project files are easier to share, audit, and version-control. When you start a new essay, copy the entries you need from previous projects.

Group entries with comment headers

BibTeX comments use %. Use them to organise:

% ---------- Anti-Oedipus and post-structuralist context ----------

@book{deleuze1977antioedipus, ... }
@misc{lacan1972parle, ... }

% ---------- Critical theory of AI and capitalism ----------

@book{pasquinelli2023eye, ... }
@book{varoufakis2024technofeudalism, ... }

Future-you, scrolling for an entry, will thank present-you.

Add TODOs for incomplete entries — don't fabricate

Sometimes you're drafting and you don't have full bibliographic info for a source. Don't make up an arXiv ID or a publisher. Add the entry with placeholder fields and a clear marker:

@article{hristova_alignment,
  title  = {[TODO: confirm full citation]},
  author = {Hristova, [TODO] and others},
  year   = {[TODO]},
  note   = {Cited in-text on alignment; full details to be supplied}
}

The [TODO] markers will appear in your rendered bibliography in glaring fashion, so you can't ship the document without fixing them. This is much safer than guessing — wrong citations are harder to catch than missing ones.

Convert prose mentions to keys when you finish a draft

While drafting, just write naturally: "Pasquinelli's Eye of the Master argues...". When you're ready to formalise, do a sweep: for each author mention, find or create the .bib entry, then replace the prose mention with the citation key. This separation keeps drafting fast.

8.Common pitfalls and how to debug them

Pitfall 1: A citation renders as (?, ?) or (Author n.d.)

You forgot to add the entry to your .bib, or the citation key doesn't match. Check spelling carefully — citation keys are case-sensitive and unforgiving.

Pitfall 2: A citation renders with missing pieces, like "Foucault, M. (1977). Preface. , xi–xiv."

This is a real one we hit recently. The entry was:

@incollection{foucault1977preface,
  title     = {Preface},
  author    = {Foucault, Michel},
  booktitle = {Anti-Oedipus: Capitalism and Schizophrenia},
  editor    = {Deleuze, Gilles and Guattari, F{\'e}lix},
  publisher = {Viking Press},
  year      = {1977},
  pages     = {xi--xiv}
}

The problem: @incollection is for a chapter in a volume edited by someone — but Deleuze and Guattari aren't editors of Anti-Oedipus, they're its authors. Some CSL templates silently drop fields they can't reconcile. Result: the booktitle and editor disappear from the output.

The fix is to restructure as @misc with the relation embedded in the title, where it's guaranteed to render:

@misc{foucault1977preface,
  author    = {Foucault, Michel},
  title     = {Preface to {Anti-Oedipus: Capitalism and Schizophrenia} by Gilles Deleuze and F{\'e}lix Guattari},
  year      = {1977},
  publisher = {Viking Press},
  pages     = {xi--xiv}
}
Now renders cleanly as:
Foucault, M. (1977). Preface to Anti-Oedipus: Capitalism and Schizophrenia by Gilles Deleuze and Félix Guattari. Viking Press.

The general principle: when a citation renders weirdly, suspect a mismatch between your entry type and the actual bibliographic relation. @misc with everything in the title is always a safe fallback.

Pitfall 3: Title case is being mangled

Some CSL styles (APA in particular) apply sentence case to titles, which can break proper nouns: Anti-Oedipus becomes Anti-oedipus. Wrap the protected text in extra braces:

title = {Preface to {Anti-Oedipus: Capitalism and Schizophrenia}},

Anything inside the inner braces won't be re-cased.

Pitfall 4: Special characters break the entry

Use UTF-8 directly: Félix, Žižek, Müller. Modern pandoc handles these without escapes. The old LaTeX escapes (F{\'e}lix) still work, but they're harder to read.

Pitfall 5: Multiple citations rendering as one

Don't write [@a, @b]. Use semicolons: [@a; @b]. The comma form is interpreted as suffix metadata for a single citation.

How to diagnose: render to plain text and inspect

When a citation looks wrong in your final HTML or PDF, render to a debugging-friendly format:

pandoc essay.md --citeproc -t plain | grep -A1 "AuthorName"

This shows you exactly what citeproc produced, with no HTML/PDF formatting in the way. If the output is still wrong, the problem is in your .bib entry. If it's right in plain text but wrong in HTML, the problem is your CSL file or output style.

Recap: the whole workflow in six lines

# 1. Write your essay in markdown with citation keys
#    e.g. Pasquinelli's argument [@pasquinelli2023eye]...

# 2. Add a YAML header pointing to your bib file
#    ---
#    bibliography: refs.bib
#    ---

# 3. Render
pandoc essay.md --citeproc --csl=chicago-author-date.csl -o essay.html

That's it. Once you've done it once, every subsequent essay is faster. Once you've done it ten times, you'll never want to format a bibliography by hand again.
